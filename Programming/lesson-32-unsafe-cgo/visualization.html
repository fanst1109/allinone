L32 — unsafe & CGo: thoát khỏi Go safety

Hai cánh cửa hậu của Go: unsafe (đo & bẻ memory layout) và cgo (gọi C). Ba module tương tác bên dưới giúp bạn thấy những thứ thường vô hình: padding trong struct, cùng địa chỉ memory đọc dưới nhiều kiểu, và overhead khi vượt biên Go↔C.

Chọn một struct ở dropdown để xem byte-by-byte memory layout: màu là field thật, cam (pad) là padding compiler chèn để giữ alignment. So sánh với phiên bản "sắp xếp lại" để thấy bớt được bao nhiêu byte.

Module 3 — Go ↔ C boundary: cgo call overhead

Animation mô phỏng các bước khi Go gọi C qua C.strlen(...): pin goroutine vào OS thread, switch register ABI, copy string, gọi C, copy ngược, unpin. Mỗi bước thêm vài ns; tổng cộng ~100–200 ns mỗi lần — gấp 100× một Go function call thường.

GO goroutine G7 đang chạy trên thread M3
GO gọi C.strlen(C.CString("hello"))
PIN pin G7 vào M3 — scheduler không thể swap G khác lên M3 nữa
GO save Go registers, switch sang C ABI (System V)
C malloc + copy "hello\0" sang C heap (CString)
C gọi strlen — duyệt đến byte zero
GO restore Go registers, return value 5 về Go
UNPIN unpin G7 — M3 lại sẵn sàng chạy G khác
GO ⚠ cs vẫn ở C heap — phải C.free tay (chưa làm = leak!)

💡 Hệ quả thực tế: hot loop gọi C.something() 1 triệu lần/giây → 150 ms CPU/giây = 15% một core, chỉ riêng overhead vượt biên. Đó là lý do Go std lib viết MD5, SHA, JSON, regex bằng pure Go — không wrap OpenSSL/PCRE qua CGo.
